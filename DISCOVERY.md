## 1. Remote/Multi-Tenant Completeness

**VERDICT: the `remote` deployment mode is fully implemented in this repo. It does not proxy org/issue/invitation logic to a closed cloud service.** The only external/private dependency is an optional billing crate (Stripe), which is feature-gated and strippable.

### 1.1 `RemoteClient` is the desktop's HTTP client, not a proxy

`crates/services/src/services/remote_client.rs:142`:
```rust
pub struct RemoteClient {
    base: Url,
    http: Client,
    auth_context: AuthContext,

```
It issues REST calls (with retry/backoff and JWT refresh) to a configurable `base` URL — and that URL is *this repo's* remote server. Endpoints it calls (all implemented server-side in `crates/remote/`):

## 2. Authentication & Identity

### 2.1 Remote mode: a complete auth stack lives in this repo

| Component | File | Notes |
|---|---|---|
| JWT (HS256) | `crates/remote/src/auth/jwt.rs` | `AccessTokenClaims {sub, session_id, iat, exp, aud}`; access TTL = **120s**; refresh TTL = **365 days** |
| Auth middleware | `crates/remote/src/auth/middleware.rs:35-172` | `require_session` validates `Authorization: Bearer`; session lookup against PG `auth_sessions`, checks `revoked_at` + inactivity |
| Sessions | `crates/remote/src/db/auth.rs:34-111` | `AuthSessionRepository` create/get/touch/rotate; `AuthSessionError::TokenReuseDetected` = refresh-token-theft detection |
| OAuth 2.0 + PKCE | `crates/remote/src/auth/provider.rs`, `handoff.rs` | `GitHubOAuthProvider`, `GoogleOAuthProvider`; web handoff with CSRF state (len 48), 10-min handoff TTL |
| OAuth routes | `crates/remote/src/routes/oauth.rs` | public: `/auth/methods`, `/auth/local/login`, `/oauth/web/{init,redeem}`, `/oauth/{provider}/{start,callback}`; protected: `/profile`, `/oauth/logout` |
| Client JWT | `shared/jwt.ts:22-26` | refresh leeway 20s, `aud='access'` |

Config (`crates/remote/src/config.rs`): `VIBEKANBAN_REMOTE_JWT_SECRET` (≥32 chars, required), `GITHUB_OAUTH_CLIENT_ID/SECRET`, `GOOGLE_OAUTH_CLIENT_ID/SECRET`, `SELF_HOST_LOCAL_AUTH_EMAIL/PASSWORD`. At least one auth method required.

### 2.2 Local "self-host" credential — single shared secret, plaintext compare

`crates/remote/src/auth/local.rs:42-55` (verified directly):
```rust
let Some(local_auth) = state.config().auth.local() else { return Err(LocalAuthError::Disabled) };
let normalized_email = local_auth.email().trim().to_ascii_lowercase();
if payload.email.trim().to_ascii_lowercase() != normalized_email
    || payload.password != local_auth.password().expose_secret()
{ return Err(LocalAuthError::InvalidCredentials); }
```
**Important nuance:** this is **not a per-user password table** — it's a single operator credential read from env (`SELF_HOST_LOCAL_AUTH_EMAIL/PASSWORD`), compared in plaintext. There is no bcrypt/argon2 because there are no stored user password hashes at all. On success it upserts a user row and ensures a personal org. For a multi-user product you would add a real credential store + hashing; today this is a single-tenant self-host shortcut, gated by `state.config().auth.local().is_some()`.

### 2.3 Local server (the desktop/CLI app): zero app-user auth

`crates/server/src/routes/mod.rs:36-68` wraps all relay routes with `middleware::require_relay_request_signature`. That middleware (`crates/server/src/middleware/relay_request_signature.rs:30-69`) is **transport integrity/origin verification** (SPAKE2-derived signing session, timestamp, nonce, signature) — not user identity. No login, no JWT, no session for local users. The local server is effectively open on localhost; security comes from the relay signing layer, not from authenticating a human.

### 2.4 `trusted-key-auth` crate = crypto primitives, not user auth

`crates/trusted-key-auth/src/lib.rs` exposes `spake2`, `request_signature`, `key_confirmation`, `refresh`, `trusted_keys`, `runtime`, `error`. Used for relay pairing/signing, not passwords.

### 2.5 GitHub: two distinct systems (do not conflate)

- **GitHub OAuth** (app *user* login) — `GitHubOAuthProvider`, env `GITHUB_OAUTH_CLIENT_ID/SECRET`.
- **GitHub App** (webhooks / app→GitHub API, e.g. PR reviews) — env `GITHUB_APP_ID/PRIVATE_KEY/WEBHOOK_SECRET/SLUG`; installs stored in `crates/remote/migrations/20251215000000_github_app_installations.sql`. RS256 app JWT, not user identity.
- **Git CLI auth** (PAT/SSH for clone/push): NOT FOUND in repo — delegated to the user's local git config / `gh`.

### 2.6 Gap analysis — to add real multi-user auth

Exists (remote): JWT, sessions, refresh rotation + reuse detection, OAuth PKCE (GitHub/Google), Bearer middleware, membership checks, relay signing.
Must add for a real user base: **per-user credential store + password hashing** (argon2/bcrypt), **rate limiting** on `/auth/local/login` + OAuth callback + refresh (NOT FOUND), **email verification / password reset** (NOT FOUND), **account lockout** (NOT FOUND), optional **MFA** (NOT FOUND), and (verify) provider-token-at-rest encryption (`encrypted_provider_tokens` table exists). For **local mode** an auth layer would be net-new from zero — there is no user concept there at all.

## 4. Task/Attempt & Assignment Surface

### 4.1 Lifecycle: Task → Workspace → Session → ExecutionProcess

The legacy `task_attempts` table was refactored away (`crates/db/migrations/20251216142123_refactor_task_attempts_to_workspaces_sessions.sql`). Current chain:
- **Task** — `crates/db/src/models/task.rs:23-33`; `tasks` table from `migrations/20250617183714_init.sql:12-22` (`id, project_id, title, description, status, parent_workspace_id, created_at, updated_at`).

- **Session** (binds an executor to a workspace) — `crates/db/src/models/session.rs:27` `executor: Option<String>`; `update_executor` at `:215-228`.
- **ExecutionProcess** (one run) — `crates/db/src/models/execution_process.rs`; carries `run_reason` + `executor_action` (an `ExecutorActionType`, e.g. `CodingAgentInitialRequest(ExecutorConfig)`).

Flow: `POST` create+start → `crates/server/src/routes/workspaces/create.rs:212` `create_and_start_workspace` (request carries `executor_config`, `:220`) → workspace row (`:237`) → `crates/services/src/services/container.rs` `start_workspace` creates the `Session` with `executor: Some(executor_config.executor.to_string())` (`container.rs:1063-1072`) → `ExecutionProcess::create` stores the `executor_action` (`container.rs:~1173-1179`) → executor spawned.

### 4.2 Where "assignment" happens today

**It doesn't — there is no assignment engine.** The executor is whatever the HTTP client sent in `executor_config`. The decision point is literally `container.rs:1066`. This is the seam to intercept.

### 4.3 Cleanest insertion points (additive, minimal upstream edits)

1. **`complexity_tier` on Task** — new migration `crates/db/migrations/<ts>_task_complexity_tier.sql` adding `complexity_tier TEXT DEFAULT 'standard' CHECK (...)`; add the field to `crates/db/src/models/task.rs` Task struct. (SQLx migrations auto-discover; no registration edit.)

3. **Assignment engine** — best kept as a **new crate** `crates/assignment-engine/` (deps: `db`, `executors`, `services`) *or* a new service module `crates/services/src/services/assignment_service.rs` (one-line `mod.rs` edit). Hook it in at `container.rs:1063` *before* `Session::create`: resolve the Task, read `complexity_tier`, pick an `Agent` where `task.complexity_tier <= agent.max_tier` AND available AND task unassigned+unblocked, scoped to a sprint; override `executor_config.executor`. The existing `executor_config` becomes the fallback/override, so the HTTP API stays backward-compatible.

The single unavoidable upstream edit for wiring is the ~one-line call inserted at `container.rs:1063`; everything else is new files + additive migrations.

## Top 5 Risks / Open Questions

1. **Auth is the load-bearing gap.** Remote has OAuth+JWT+sessions, but there is **no hashed multi-user credential store** (local login = one plaintext env secret, `crates/remote/src/auth/local.rs:52`) and **local mode has no app-user auth at all**. Any multi-tenant product needs this built fresh, plus rate limiting / lockout / email-verify / reset (all NOT FOUND).
2. **Branding changes that break compatibility.** SPAKE2 ID constants (`crates/relay-client/src/lib.rs`), Tauri identifier `ai.bloop.vibe-kanban`, and the BloopAI releases URL (`releases.rs:14`) are not cosmetic — changing them breaks relay pairing, desktop auto-update, and update checks respectively. Sequence the rebrand deliberately.
3. **Private billing dependency.** `crates/remote/Cargo.toml:17` pulls `billing` from `ssh://git@github.com/BloopAI/vibe-kanban-private` (feature `vk-billing`). A fork cannot build that feature; confirm it stays stripped (`FEATURES` empty in Dockerfile) or replace it.
4. **Assignment seam is narrow but real.** The only place executor selection happens is `container.rs:1066`, fed straight from the HTTP request. The insertion is clean, but the Task↔Workspace↔Session model recently changed (attempts→sessions, `20251216142123`); confirm `parent_workspace_id`/sprint scoping semantics before wiring tier-based pickup, and decide whether assignment lives in `db`-backed `Agent` rows vs executor profiles.
5. **Remote standalone has heavy operational surface.** Running Postgres mode needs Postgres (`wal_level=logical`) + ElectricSQL + JWT secret + ≥1 OAuth provider + (optionally) R2/Azure/Loops/Stripe, with reverse proxy external to compose (`Caddyfile.example` not wired in). Verify the minimum viable subset (DB + Electric + one auth provider) actually boots before committing to it as the fork's backbone.
