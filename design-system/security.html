Trust & isolation

Security model

Coding agents run real commands on real machines. Tasca treats untrusted, externally-filed work as a first-class threat — and gates it before any of it can reach a shell.

Untrusted input can't execute

External clients propose and comment. They never trigger a run. An internal human must promote a ticket to agent-ready first.

Every external run is sandboxed

Ephemeral container per run — no host filesystem, no secrets, restricted egress. Other clients' repos are never visible.

Secrets encrypted at rest

Passwords are argon2id. Provider tokens and org keys are AES-256-GCM under a dedicated server key — never the JWT secret.

01 Threat model

Tasca dispatches coding agents that run shell commands, edit files, and open pull requests. Many agents default to permissive execution. That is acceptable for trusted, internal work — and dangerous the moment untrusted input enters the system.

The primary risk is a ticket authored or influenced by an external party that causes an agent to execute attacker-controlled instructions on a host with access to source, secrets, or the network. Tasca's security model is built to make that path impossible by default.

02 Trust tiers

Identity and roles live in the team layer, and that is where the gate is enforced — before any dispatch reaches a local executor. It cannot be bypassed at the execution seam.

  • External / guest is propose-only. Guests file and comment. They cannot start a run.
  • Promotion is explicit. An internal member promotes a proposed ticket to agent-ready. Untrusted input never directly triggers execution.
  • Local-only mode (no team layer) is treated as fully trusted and internal — there is no external surface to gate.

03 Sandboxed execution

Any external-touch execution runs inside an ephemeral container, provisioned with only the client's worktree and toolchain. The run has:

  • No host filesystem and no host secrets.
  • Restricted egress, scoped to what the task legitimately needs.
  • No visibility into other clients' repositories.

The model host and GPU stay on the host machine; the sandbox reaches inference over the network. Permissive agent defaults are flipped to supervised / approval-gated on any non-trusted path.

04 Prompt-injection containment

Tasca can re-dispatch an agent with PR review comments injected as a follow-up prompt. Because review text is attacker-reachable on external work, that behavior is off by default for any ticket carrying external comments. Review feedback never silently becomes an instruction the agent obeys.

05 Secrets & auth

Credentials are never stored in plaintext.

  • Passwords are hashed with argon2id.
  • Provider tokens and org API keys are encrypted with AES-256-GCM, keyed by a dedicated SERVER_ENCRYPTION_KEY — decoupled from the JWT secret, so rotating JWTs can't orphan stored keys.
  • Sessions use short-lived access tokens with refresh-token rotation and reuse detection; a detected reuse revokes the whole session family.
  • Login, refresh, and OAuth callbacks are rate-limited, with account lockout after repeated failures.

The org Anthropic key powers the PM-assistant only — never worker agents — and is validated on entry, masked on read, and excluded from every log and audit payload.

06 Roles & access

Four roles, each a strict superset of the next. The guest tier is the security boundary for external collaboration.

RoleCan do
ownerBilling, org keys, delete org, plus everything an admin can do.
adminManage members, projects, agents, sprints; set the org key; promote proposals.
memberCreate and assign tickets, review PRs, run agents.
guestPropose-only: file and comment. Cannot trigger execution.

07 Hosting & data isolation

On hosted Tasca, each organization's data is isolated per tenant and encrypted in transit and at rest. We do not use your code, tickets, or prompts to train models. The product bundles no third-party analytics or crash reporting — the only data we process is what the service needs to run, described in our privacy policy.

You can export or delete your organization's data at any time. And because Tasca's core is open source under Apache-2.0, you can move to a self-hosted deployment whenever you want — the product is identical.

08 Responsible disclosure

If you find a vulnerability, please report it privately before disclosing publicly. Email security@tasca.dev with steps to reproduce and the affected version. We aim to acknowledge within two business days and to coordinate a fix and disclosure timeline with you.

Because Tasca is self-hosted, operators are responsible for patching their own deployments. Security-relevant releases are tagged and noted in the changelog.
